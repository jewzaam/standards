# .NET Project Structure

Applies to all C# projects. Plugin/host-specific csproj requirements
(install paths, deploy targets, host-shipped assembly exclusions) are
covered by the relevant subsystem standard — for NINA, see
[nina-plugin.md](nina-plugin.md).

## Solution and project layout

```text
<repo-root>/
├── <Project>.sln
├── <Project>.csproj            ← main library / executable
├── Properties/
│   └── AssemblyInfo.cs         ← when manifest metadata is read from attributes
├── <feature folders>/
├── tests/
│   └── <Project>.Tests.csproj
├── packages.lock.json          ← committed; deterministic restore
├── CHANGELOG.md
├── README.md
└── LICENSE
```

- One csproj per assembly. Multiple projects belong in a `src/` /
  `tests/` split only when there are 3+ projects; below that, a flat
  layout is clearer.
- Commit `packages.lock.json`. Set
  `<RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>` so CI
  fails on lock drift.
- Do not commit `bin/`, `obj/`, `*.user` files. Standard
  [dotnet `.gitignore`](https://github.com/github/gitignore/blob/main/VisualStudio.gitignore)
  template is sufficient.

## Target framework selection

- **Class libraries with no platform dependency**: `net8.0`.
- **WPF / Windows-only**: `net8.0-windows` (or whatever the host pins —
  e.g., NINA 3.2 pins plugins to `net8.0-windows`).
- **Cross-platform CLI tools**: `net8.0`.
- Use the lowest TFM that satisfies dependency and host constraints.
  Bumping the TFM is a versioned change.

Multi-targeting (`<TargetFrameworks>net8.0;net8.0-windows</TargetFrameworks>`)
is acceptable when the same code legitimately runs both places. Avoid it
when one target is a workaround for tooling rather than a real
constraint.

## Baseline csproj

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net8.0</TargetFramework>
    <Nullable>enable</Nullable>
    <LangVersion>latest</LangVersion>
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
    <RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>
  </PropertyGroup>
</Project>
```

Add when applicable:

- `<UseWPF>true</UseWPF>` — required for any project that includes XAML.
- `<OutputType>Library</OutputType>` — class libraries (default for
  `Microsoft.NET.Sdk` is library, but state it explicitly in plugin
  projects where the loader cares).
- `<Platforms>x64</Platforms>` + `<PlatformTarget>x64</PlatformTarget>` —
  when the host is 64-bit only (NINA, most desktop hosts).
- `<CopyLocalLockFileAssemblies>true</CopyLocalLockFileAssemblies>` —
  when deploying with `dotnet build` outputs a folder that other tooling
  picks up (e.g., a plugin folder).

## Dependencies

- Prefer a single top-level `PackageReference` to a meta-package or
  framework-supplied bundle (e.g., NINA's `NINA.Plugin`) and rely on
  transitive resolution.
- Pin direct dependencies; let transitive ones float within their
  declared range.
- When a host already ships a third-party DLL (CommunityToolkit.Mvvm,
  Newtonsoft.Json, Serilog, etc.), reference it with
  `<ExcludeAssets>runtime</ExcludeAssets>` so the build picks up
  compile-time symbols without publishing the runtime DLL:

```xml
<PackageReference Include="CommunityToolkit.Mvvm" Version="8.4.0">
  <ExcludeAssets>runtime</ExcludeAssets>
</PackageReference>
```

- For framework-bundled meta-packages such as `NINA.Plugin`, use
  `<PrivateAssets>all</PrivateAssets>` to keep their transitively-pulled
  DLLs out of the publish folder.

## Versioning

- Follow [SemVer](https://semver.org/). New projects start at `0.1.0`.
- `AssemblyVersion` and `AssemblyFileVersion` use 4-segment
  `M.m.p.b` format. Each segment is a 16-bit unsigned integer
  (`0`–`65535`) per CLR.
- A version string of `"1.0"` is malformed at the assembly-attribute
  layer — always emit 4 segments even when the trailing segments are
  zero.

## Generated file headers

Every source file generated by an AI tool carries the attribution line
at the top of the file:

```csharp
// Generated By: Claude Code (Claude Opus 4.7)
```

Human-authored files do not require this header. Files mixed-authored
(human edits to AI-generated scaffold) keep the header until the file is
substantively rewritten by hand.
